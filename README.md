# Утилита обновления прошивок YADRO Vegman Rx20 (G1-3)

CLI-утилита для обновления прошивок **BMC** и/или **UEFI** серверов YADRO Vegman Rx20
(платформа на базе OpenBMC). Утилита заливает образ прошивки напрямую в BMC по Redfish
**HTTP-push** (`POST` в `HttpPushUri`, модель push — оператор отправляет образ сам),
отслеживает установку через Redfish `TaskService` и `FirmwareInventory`, при необходимости
включает сервер и печатает итоговую сводку. Работает только по Redfish (HTTPS, Basic auth).

## Требования к окружению

- **ОС:** Linux (целевая — Alt Linux 10.2; работает на любом современном дистрибутиве).
- **Python:** 3.9+ (на целевом хосте используется встроенный Python 3.9.25).
- **Зависимости:** отсутствуют. Используется только стандартная библиотека — `pip install`
  не требуется.
- **Права:** обычный пользователь. Локальный порт утилита не слушает, `sudo`/root и
  capability `CAP_NET_BIND_SERVICE` **не нужны**.
- **Сеть:** только хост оператора → BMC по TCP 443 (Redfish, HTTPS). Обратный доступ
  BMC → хост оператора и локальный HTTP-сервер раздачи больше не требуются — образ
  отправляется в BMC по тому же соединению (HTTP-push).

> **TLS:** сертификат BMC **не проверяется** (на оборудовании CA доверен внутри ОС, но на
> части устройств встречаются ошибки применения сертификата). Это поведение фиксировано и
> не управляется флагами.

## Состав каталога

```
Updater/
├── updater.py            # исполняемый файл утилиты
├── firmware/
│   ├── bmc/   <образ прошивки BMC>      # ровно один файл образа
│   └── uefi/  <образ прошивки UEFI>     # ровно один файл образа
├── README.md
├── ruff.toml             # конфигурация линтинга (для разработки)
├── .gitignore
└── logs.txt              # журнал (создаётся при запуске; путь задаётся --log-file)
```

Положите образ прошивки BMC в `firmware/bmc/`, образ UEFI — в `firmware/uefi/`. В каждом
каталоге должен лежать **ровно один** файл образа (имя произвольное); назначение прошивки
(BMC/UEFI) определяется самим образом. Пути ищутся относительно расположения `updater.py`,
поэтому утилиту можно запускать из любого рабочего каталога.

## Запуск

```bash
python3 updater.py --host <fqdn|ip> (--all | --bmc | --uefi) \
    --login admin --password secret [опции]
```

> **Важно:** перед запуском сервер должен быть **выключен** (`PowerState: Off`). Если сервер
> включён, утилита прекратит работу с ошибкой, не выполняя обновление.

### Примеры

```bash
# Обновить и BMC, и UEFI, затем включить сервер:
python3 updater.py --host bmc01.example.com --all --login admin --password secret

# Обновить только BMC:
python3 updater.py --host 10.0.0.5 --bmc --login admin --password secret

# Обновить только UEFI, не включать сервер после обновления:
python3 updater.py --host 10.0.0.5 --uefi --no-autoboot --login admin --password secret

# Тестовый прогон без обновления (только сбор информации, без HTTP-push):
python3 updater.py --host 10.0.0.5 --all --dry-run --login admin --password secret

# Увеличенный таймаут заливки/применения для медленного канала:
python3 updater.py --host 10.0.0.5 --bmc --update-timeout 3600 \
    --login admin --password secret
```

## Флаги

| Флаг | Обяз. | По умолчанию | Назначение |
|------|:----:|--------------|------------|
| `--host` | да | — | FQDN или IP-адрес BMC сервера. |
| `--all` | да¹ | — | Обновить и BMC, и UEFI (UEFI, затем BMC; в конце — перезагрузка BMC). |
| `--bmc` | да¹ | — | Обновить только BMC. |
| `--uefi` | да¹ | — | Обновить только UEFI. |
| `--login` | да | — | Имя пользователя BMC (например, `admin`). |
| `--password` | да | — | Пароль пользователя BMC. |
| `--no-autoboot` | нет | выкл. | НЕ включать **хост** после обновления (по умолчанию включается, `ResetType: On`). На обязательную перезагрузку BMC после его обновления **не влияет**. |
| `--log-file` | нет | `logs.txt` | Путь к файлу журнала. |
| `--update-timeout` | нет | `1800` | Таймаут заливки образа (HTTP-push) и его применения/подтверждения, секунды. |
| `--online-timeout` | нет | `900` | Таймаут ожидания возврата BMC в строй после перезагрузки, секунды. |
| `--dry-run` | нет | выкл. | Только сбор информации; без HTTP-push и reset. |

¹ Ровно один из взаимоисключающих флагов `--all` / `--bmc` / `--uefi` обязателен.

## Что делает утилита (порядок работы)

1. Сбор информации о сервере по Redfish (`/redfish/v1/Systems/system`,
   `FirmwareInventory/bios_active`, `FirmwareInventory/bmc_active`) и **проверка, что сервер
   выключен** (`PowerState: Off`).
2. Получение `HttpPushUri` из `/redfish/v1/UpdateService`.
3. Заливка образа(ов) HTTP-push'ем: `POST` бинарника в `HttpPushUri`
   (`Content-Type: application/octet-stream`, тело — файл образа) → опрос созданной задачи в
   `/redfish/v1/TaskService/Tasks/<id>` до `TaskState: Completed`. При `--all` порядок —
   **сначала UEFI, затем BMC** (BMC активирует образ без авто-перезагрузки, поэтому
   обновляется последним).
4. Завершение:
   - включение сервера (`ComputerSystem.Reset` с `{"ResetType": "On"}`), если не задан
     `--no-autoboot`; на старте применяется обновлённая прошивка UEFI;
   - если обновлялся BMC — **обязательная перезагрузка BMC** (`Manager.Reset` с
     `{"ResetType": "GracefulRestart"}`) с ожиданием возврата в строй (`--online-timeout`).
     Без неё ломается провижининг ресурсов UEFI и обмен FRU-данными. Флаг `--no-autoboot`
     на перезагрузку BMC не влияет.
5. Снятие диагностики: повторный сбор информации и вывод итоговой сводки (FQDN, версии
   прошивок до/после, состояние сервера) в консоль и в журнал.

## Журналирование

Все действия и ответы сервера пишутся в файл (`--log-file`, по умолчанию `logs.txt`) и
дублируются в консоль. Секреты (пароль при логине, заголовок авторизации) в журнал не
попадают.

## Коды возврата

| Код | Значение |
|-----|----------|
| `0` | Успешное завершение. |
| `2` | Управляемая ошибка (сервер включён, активация `Failed`, таймаут, отклонённый запрос и т.п.). |
| `1` | Непредвиденная ошибка. |
| `130` | Прервано пользователем (Ctrl+C). |

## Примечания

- После обновления BMC активирует образ, но **не перезагружается сам**, поэтому по завершении
  утилита принудительно перезагружает BMC (`Manager.Reset` = `GracefulRestart`) — иначе ломается
  провижининг ресурсов UEFI и обмен FRU-данными. Перезагрузка выполняется и при `--bmc`, и при
  `--all`; флаг `--no-autoboot` её не отменяет (он гасит только включение хоста). BMC уходит в
  ребут **не мгновенно** (наблюдалась задержка ~30 c), поэтому утилита сначала дожидается
  фактического начала перезагрузки (ухода BMC в офлайн), и только затем — возврата в строй
  (`--online-timeout`). На время ребута BMC временно недоступен — утилита терпит обрывы связи
  (`Connection refused`), обрывы HTTP-ответа и `HTTP 5xx`/`404`, а сбор финальной диагностики
  устойчив к кратковременным 5xx сразу после возврата (несколько повторов).
- Образ доставляется **HTTP-push'ем** (`POST` бинарника в `HttpPushUri`,
  `application/octet-stream`), а не через `SimpleUpdate`: на этой платформе `SimpleUpdate`
  принимает только `TransferProtocol: TFTP`, который недоступен по сети (из BMC открыты
  только TCP 80/443/22, а TFTP — это UDP/69). Заливка идёт по уже доступному каналу
  оператор→BMC (443) с явным `Content-Length` (без `Transfer-Encoding: chunked`).
- Версия UEFI/BIOS в инвентаре обновляется после старта хоста, поэтому в сводке сразу после
  запуска она может отображаться ещё прежней.
- FQDN берётся из `/redfish/v1/Managers/bmc/EthernetInterfaces/eth0` (поле `FQDN`, фолбэк —
  `HostName`) и выводится в итоговой сводке.

## Разработка

Утилита не требует зависимостей для работы. Для разработки используется линтер
[`ruff`](https://docs.astral.sh/ruff/) (конфигурация — `ruff.toml`):

```bash
ruff check .          # либо: python3 -m ruff check .
python3 -m py_compile updater.py
```

Код единообразно использует `%`-форматирование строк (в т.ч. ленивое в вызовах `log.*`),
поэтому правило `pyupgrade` (UP) в линтере намеренно отключено.
