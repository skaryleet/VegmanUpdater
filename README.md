# Утилита обновления прошивок YADRO Vegman Rx20 (G1-3)

CLI-утилита для обновления прошивок **BMC** и/или **UEFI** серверов YADRO Vegman Rx20
(платформа на базе OpenBMC). Утилита раздаёт образ прошивки по HTTP, инициирует обновление
через Redfish (`SimpleUpdate`, модель pull — BMC сам скачивает образ), отслеживает активацию
через legacy REST OpenBMC, при необходимости включает сервер и печатает итоговую сводку.

## Требования к окружению

- **ОС:** Linux (целевая — Alt Linux 10.2; работает на любом современном дистрибутиве).
- **Python:** 3.9+ (на целевом хосте используется встроенный Python 3.9.25).
- **Зависимости:** отсутствуют. Используется только стандартная библиотека — `pip install`
  не требуется.
- **Права:** для HTTP-сервера на порту 80 нужны права root (`sudo`) либо capability
  `CAP_NET_BIND_SERVICE`. Можно указать непривилегированный порт через `--http-port`.
- **Сеть:**
  - хост оператора → BMC по TCP 443 (Redfish + legacy REST, HTTPS);
  - BMC → хост оператора по выбранному `--http-port` (по умолчанию 80) — BMC скачивает
    образ с нашего HTTP-сервера. IP, который попадёт в `ImageURI`, определяется
    автоматически; при несимметричной маршрутизации задайте его явно через `--advertise-ip`.

> **TLS:** сертификат BMC **не проверяется** (на оборудовании CA доверен внутри ОС, но на
> части устройств встречаются ошибки применения сертификата). Это поведение фиксировано и
> не управляется флагами.

## Состав каталога

```
Updater/
├── updater.py            # исполняемый файл утилиты
├── firmware/
│   ├── bmc/   <образ прошивки BMC>      # ровно один файл образа
│   └── uefi/  <образ прошивки UEFI>     # ровно один файл образа
├── README.md
└── logs.txt              # журнал (создаётся при запуске; путь задаётся --log-file)
```

Положите образ прошивки BMC в `firmware/bmc/`, образ UEFI — в `firmware/uefi/`. В каждом
каталоге должен лежать **ровно один** файл образа (имя произвольное); назначение прошивки
(BMC/UEFI) определяется самим образом. Пути ищутся относительно расположения `updater.py`,
поэтому утилиту можно запускать из любого рабочего каталога.

## Запуск

```bash
sudo python3 updater.py --host <fqdn|ip> (--all | --bmc | --uefi) \
    --login admin --password secret [опции]
```

> **Важно:** перед запуском сервер должен быть **выключен** (`PowerState: Off`). Если сервер
> включён, утилита прекратит работу с ошибкой, не выполняя обновление.

### Примеры

```bash
# Обновить и BMC, и UEFI, затем включить сервер:
sudo python3 updater.py --host bmc01.example.com --all --login admin --password secret

# Обновить только BMC:
sudo python3 updater.py --host 10.0.0.5 --bmc --login admin --password secret

# Обновить только UEFI, не включать сервер после обновления:
sudo python3 updater.py --host 10.0.0.5 --uefi --no-autoboot --login admin --password secret

# Тестовый прогон без обновления (сбор инфо + проверка раздачи образа):
sudo python3 updater.py --host 10.0.0.5 --all --dry-run --login admin --password secret

# Непривилегированный порт раздачи (без sudo) и явный advertise-IP:
python3 updater.py --host 10.0.0.5 --all --http-port 8080 --advertise-ip 10.0.0.100 \
    --login admin --password secret
```

## Флаги

| Флаг | Обяз. | По умолчанию | Назначение |
|------|:----:|--------------|------------|
| `--host` | да | — | FQDN или IP-адрес BMC сервера. |
| `--all` | да¹ | — | Обновить и BMC, и UEFI (BMC, затем UEFI). |
| `--bmc` | да¹ | — | Обновить только BMC. |
| `--uefi` | да¹ | — | Обновить только UEFI. |
| `--login` | да | — | Имя пользователя BMC (например, `admin`). |
| `--password` | да | — | Пароль пользователя BMC. |
| `--no-autoboot` | нет | выкл. | НЕ включать сервер после обновления. По умолчанию сервер включается (`ResetType: On`). |
| `--log-file` | нет | `logs.txt` | Путь к файлу журнала. |
| `--http-port` | нет | `80` | Порт HTTP-сервера раздачи образа. |
| `--advertise-ip` | нет | автоопределение | IP в `ImageURI` (адрес, по которому BMC скачивает образ). |
| `--download-timeout` | нет | `600` | Таймаут ожидания готовности образа на BMC, секунды. |
| `--activation-timeout` | нет | `1800` | Таймаут ожидания завершения активации, секунды. |
| `--dry-run` | нет | выкл. | Только сбор информации и раздача образа; без `SimpleUpdate`, активации и reset. |

¹ Ровно один из взаимоисключающих флагов `--all` / `--bmc` / `--uefi` обязателен.

## Что делает утилита (порядок работы)

1. Сбор информации о сервере по Redfish (`/redfish/v1/Systems/system`,
   `FirmwareInventory/bios_active`, `FirmwareInventory/bmc_active`) и **проверка, что сервер
   выключен** (`PowerState: Off`).
2. Поднятие HTTP-сервера, раздающего **только** выбранный образ (или два при `--all`).
3. Для каждого компонента: запуск Redfish `UpdateService.SimpleUpdate` с `ImageURI` на наш
   HTTP-сервер → ожидание готового образа (`Activation: Ready`) в
   `/xyz/openbmc_project/software/<id>` → запуск активации
   (`PUT .../attr/RequestedActivation = ...Active`) → опрос статуса раз в секунду до
   `Activation: Active`. При `--all` сначала BMC, затем UEFI.
4. Если не задан `--no-autoboot` — включение сервера (`ComputerSystem.Reset` с
   `{"ResetType": "On"}`); на старте применяется обновлённая прошивка UEFI.
5. Повторный сбор информации и вывод итоговой сводки (версии прошивок до/после, состояние
   сервера) в консоль и в журнал.

## Журналирование

Все действия и ответы сервера пишутся в файл (`--log-file`, по умолчанию `logs.txt`) и
дублируются в консоль. Секреты (пароль при логине, заголовок авторизации) в журнал не
попадают.

## Коды возврата

| Код | Значение |
|-----|----------|
| `0` | Успешное завершение. |
| `2` | Управляемая ошибка (сервер включён, активация `Failed`, таймаут, отклонённый запрос и т.п.). |
| `1` | Непредвиденная ошибка. |
| `130` | Прервано пользователем (Ctrl+C). |

## Примечания

- При обновлении BMC контроллер может временно стать недоступен (перезагрузка). Утилита
  устойчива к кратковременным обрывам связи на этапе опроса активации: ловит сетевые ошибки,
  переустанавливает сессию и продолжает опрос до таймаута.
- Запуск новой прошивки BMC на некоторых платформах может требовать отдельной перезагрузки
  BMC — это вне описанного здесь потока обновления.
- Версия UEFI/BIOS в инвентаре обновляется после старта хоста, поэтому в сводке сразу после
  запуска она может отображаться ещё прежней.
